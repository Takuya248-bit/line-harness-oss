# IG フィード投稿自動化 v4 設計書

## 概要

バリリンガルのInstagramフィード投稿を週次PDCAサイクルで自動運用する。
KPIベースのA/Bテスト自動最適化により、LINE登録数を3ヶ月で30件/月→100件/月に引き上げる。
将来リール動画も同一パイプラインで運用する拡張性を持たせる。

## 背景・課題

現行ig-auto-poster v3の問題点:
- 画像: Serper/Unsplashのストック写真で「バリ感」が薄い
- キャプション: ハードコードのフック文+テンプレ構成で単調
- フォーマット: バズフォーマット10パターンがNotion DBにあるが、実装はspot_listとhook_factsの2パターンのみ
- デザイン: Satori SVGの表現力に限界
- ネタ: Notion知識DBへの自動蓄積フローがない
- KPI: 計測・最適化の仕組みがない

## KPI設計

### 北極星KPI

LINE登録数/月（IG経由）

### 目標

| 期間 | LINE登録/月 | 備考 |
|------|-----------|------|
| 現状 | ~30件 | フォロワー1,700人 |
| Month 1 | 50件 | ファネル計測基盤+保存率改善 |
| Month 2 | 75件 | A/Bテスト本格稼働+ボトルネック解消 |
| Month 3 | 100件 | 勝ちパターン蓄積+リール追加検討 |

### ファネル4層KPI

| 層 | 指標 | 目標値 | A/Bテスト対象 |
|----|------|--------|-------------|
| 認知 | リーチ数 | 670/投稿 | 投稿時間帯、ハッシュタグ |
| 評価 | 保存率(saves/reach) | ≥3% | フォーマット、デザイン |
| 興味 | プロフ訪問率 | ≥5% | CTA文言、フック文 |
| 行動 | LINE登録数 | 100件/月 | プロフィール導線 |

### 逆算ロジック

```
LINE登録 100件/月
÷ プロフ→LINE転換率 10%
= プロフ訪問 1,000件/月
÷ プロフ訪問率 5%
= 投稿接触 20,000件/月
÷ 30投稿/月
= 1投稿あたりリーチ 670
```

フォロワー1,700人の約40%到達で現実的な水準。

## 投稿方針

- 頻度: 1日1投稿、週7日
- 形式: テキスト+イラスト主体のカルーセル（8-10枚）
- サイズ: 1080x1350px（4:5縦型、画面占有率最大）
- 内容: バリ島Tips・英語学習Tipsで価値提供→ソフトCTAでLINE登録誘導
- 直接宣伝はしない（decisions.md 2026-04-02 準拠）

### カルーセル構成（リサーチ結果準拠）

| スライド | 役割 | 設計 |
|---------|------|------|
| 1枚目（表紙） | フック | 8-10語の太字フック文、高コントラスト配色、情報ギャップ |
| 2-8枚目（中身） | 1スライド1ポイント | 見出し+2-3行説明+アイコン1点 |
| 9枚目（まとめ） | 総括 | リスト形式で全ポイント一覧 |
| 10枚目（CTA） | 行動喚起 | 「保存して」「友達に送って」「プロフのLINEから」 |

## 全体フロー

```
毎週月曜 GH Actions:

  06:00 ネタ収集バッチ
    RSS収集 → バズ投稿収集 → 抽象化 → 具体化 → Notion投入

  06:30 週次レポート生成
    先週7投稿のインサイト収集（IG Graph API）
    → A/Bテスト勝ち判定
    → ボトルネック層判定
    → 勝ちパターンDB更新
    → レポートをLINE通知

  07:00 今週7投稿バッチ生成
    ネタ選定（Notion知識DB）
    → 構成生成（Groq）
    → メディア生成（sharp）
    → キャプション生成（Groq）
    → R2保存
    → プレビュー一覧をLINE通知

  オーナー承認待ち（LINE上で一括OK）

  承認後:
    7投稿をスケジュールキューに登録

  毎日18:00 Cron:
    キューから1投稿取得 → IG API投稿
```

## A/Bテスト設計

### テスト枠

| 枠 | 投稿数/週 | 役割 |
|----|----------|------|
| コントロール群 | 5投稿 | 現在の勝ちパターン |
| テスト群 | 2投稿 | 1変数だけ変えたもの |

### テスト軸（1週間に1変数のみ）

| 軸 | テスト内容例 |
|----|------------|
| デザイン | 背景色、フォント、レイアウト |
| フォーマット | 知ってた系 vs ランキング系 etc |
| フック文 | 表紙のキャッチコピーパターン |
| 投稿時間帯 | 18:00 vs 12:00 vs 21:00 |
| ハッシュタグ | セット比較 |

### 判定指標（優先順）

1. 保存率（saves / reach）
2. シェア率（shares / reach）
3. リーチ数

いいね数は判定に使わない（相関が弱い）。

### 勝ち判定ロジック

テスト群の保存率 > コントロール群の平均 → 勝ち → 翌週コントロール群に昇格。
負けたパターンは記録して再テストしない。

### ボトルネック自動判定

システムが自動で「今どの層がボトルネックか」を判定し、そこに対するテスト軸を優先実行する。

```
保存率 < 3%           → 評価層 → デザイン/フォーマットをテスト
保存率 ≥ 3% かつ
  プロフ訪問率 < 5%    → 興味層 → CTA/フック文をテスト
プロフ訪問率 ≥ 5% かつ
  LINE登録 < 目標      → 行動層 → プロフィール改善を提案
全層OK               → 認知層 → 投稿時間帯/ハッシュタグをテスト
```

## ネタ自動蓄積パイプライン

### 収集フロー（週次、バッチ生成前に実行）

```
① RSSフィード収集
  - バリ島観光系ブログ
  - 留学・英語学習系メディア
  - インドネシア現地ニュース
      ↓
② バズ投稿収集
  - IG Discovery API（ハッシュタグ上位投稿のテーマ抽出）
      ↓
③ 抽象化（Groq）
  記事/投稿の核心テーマを抽出
  例: 「バリの隠れカフェ紹介」→「地元民しか知らない」
      ↓
④ 具体化（Groq）
  抽象テーマ × バリリンガルの一次情報 → 独自ネタに変換
  例: 「地元民しか知らない」→「うちの生徒が通う穴場3店」
      ↓
⑤ Notion知識DB投入
  - category/tags自動分類
  - source: "auto_research"
  - reliability: "unverified"
  - 重複チェック（類似タイトル）
```

### reliability別の投稿ネタ採用優先度

| reliability | 説明 | 優先度 |
|------------|------|--------|
| firsthand | 一次情報（実体験・生徒の声） | 最優先 |
| verified | 確認済み情報 | 通常 |
| unverified | 自動収集 | 勝ちフォーマットと組み合わせ、承認時に内容チェック |

一次情報の投入は従来通りknowledge-save.mjsで手動継続。

## 共通パイプラインアーキテクチャ

フィードもリールも同一パイプラインで動作する設計。

```
ContentType ("carousel" | "reel")
      ↓
ネタ選定（Notion知識DB → カテゴリ/タグ取得）
      ↓
構成生成（Groq: フック文+本文+CTA）
      ↓
メディア生成（タイプ別分岐）
  carousel → sharp画像 × 8-10枚
  reel     → Pexels動画 + edge-tts + sharpテロップ合成
      ↓
キャプション生成（Groq: フック+本文+CTA+ハッシュタグ）
      ↓
R2保存（メディア+メタデータ）
      ↓
スケジュールキュー登録（D1: 投稿日時+A/Bテストメタデータ）
      ↓
Cron投稿（IG API）
```

### A/Bテストメタデータ（各投稿に付与）

```json
{
  "content_type": "carousel",
  "test_week": "2026-W15",
  "test_axis": "design",
  "test_variant": "dark_bg_v2",
  "is_control": false
}
```

## 技術構成

### 実行環境

| 環境 | 用途 |
|------|------|
| GH Actions | 週次バッチ（ネタ収集+レポート+生成） |
| CF Worker | Cron投稿+承認API+インサイト収集 |
| CF D1 | スケジュールキュー、A/Bテスト結果、テスト履歴 |
| CF R2 | 生成画像/動画ストック |
| Notion | 知識DB、バズフォーマットDB |

### 外部API（全て無料）

| API | 用途 |
|-----|------|
| Groq (Llama 3.3-70B) | 構成生成、キャプション、抽象化/具体化 |
| Pexels | 動画素材（リール用） |
| edge-tts | ナレーション音声（リール用） |
| IG Graph API | 投稿+インサイト取得 |
| LINE Messaging API | 通知+承認 |

### ローカルツール

| ツール | 用途 |
|-------|------|
| sharp | 画像生成（テキスト+背景+アイコン） |
| ffmpeg | リール動画合成（将来） |

### リポジトリ構成

```
ig-auto-poster/
  src/
    pipeline/
      neta-collector.ts      # ネタ収集（RSS+バズ投稿+抽象化+具体化）
      content-planner.ts     # 構成生成（Groq）
      media-generator.ts     # sharp画像 / リール動画生成
      caption-writer.ts      # Groqキャプション生成
      scheduler.ts           # スケジュールキュー管理
    ab-test/
      manager.ts             # テスト設計・変数選択・判定
      reporter.ts            # 週次レポート生成
    worker/
      cron-poster.ts         # Cron投稿
      approval-api.ts        # LINE承認受付
      insights.ts            # IG Graph APIインサイト収集
    batch/
      weekly-run.ts          # GH Actionsエントリポイント
```

## 週次レポート出力例

```
週次レポート 2026-W15

LINE登録: 8件（目標10件）
リーチ: 12,400（前週比 +15%）
保存率: 4.2%（前週 3.1%）← 改善
プロフ訪問率: 3.8%（前週 4.0%）← ボトルネック

A/Bテスト結果:
  テスト: フック文パターンB vs コントロール
  勝者: パターンB（保存率 5.1% vs 3.8%）→ 昇格

今週のボトルネック: 興味層（プロフ訪問率）
→ 今週のテスト軸: CTA文言
→ テスト内容: 「プロフのリンクから」→「プロフの🔗から30秒で登録」
```

## バズフォーマット（Notion Buzz Formats DB）

TTP戦略で模倣するフォーマット10パターン。content_templatesテーブルのweight値で出現頻度を制御。

1. 知ってた系 - 「〇〇って知ってた？」
2. あるある系 - 「バリ留学あるある」
3. やってみた系 - 「実際に〇〇してみた」
4. vs比較系 - 「日本 vs バリ」
5. ランキング系 - 「TOP5」
6. クイズ系 - 「これ何だと思う？」
7. 衝撃事実系 - 「実は〇〇だった」
8. Before/After系 - 「留学前と後」
9. 文化差分系 - 「日本人が驚くバリの常識」
10. ネイティブ検証系 - 「ネイティブに聞いてみた」
