# プランニングルール

## タスクフォーマット（必須）

プランやタスク分解時は「基準が先、手法が後」の順序を厳守する。

```
### タスク名
- 出自: （元の要件/ユーザー指示/調査結果のどれに基づくか）
- Done when: （完了条件。検証可能で曖昧さゼロ）
- Verified by: （検証方法。コマンド、テスト、目視確認等の具体的手段）
- Approach: （実装アプローチ。Done whenを満たすための手順）
```

必ず守ること:
- 全タスクにDone when + Verified by + Approachの3点を含める
- Done whenは検証可能な具体的条件にする（例: 「tsc --noEmitがエラー0で通る」）
- Verified byは確認対象を明示する（例: 「/api/usersのレスポンスが200を返す」）

適用範囲: TaskCreate、プラン・設計文書のタスク分解（progress.mdのログには不要）

## スコープクリープ検出

自動警告トリガー:
1. サブタスク数が15を超えた場合 → 「スコープが膨張していませんか？」と警告 → フェーズ分割を提案
2. 元の要件に遡れないサブタスクが存在する場合 → 「出自が不明です。元の要件のどれに対応しますか？」と確認

スコープ判定基準:
- 必須: ユーザーが明示的に依頼したもの
- 推奨: 依頼を満たすために技術的に必要なもの
- 削除候補: 「ついでにやっておいた方がいい」レベルのもの → ユーザーに提示して判断を仰ぐ

## プラン検証

以下の条件を満たすプランのみ実施（軽微な修正・単一ファイル変更は対象外）:
- 新機能追加 / アーキテクチャへの影響 / 外部API新規利用

条件を満たす場合、dev-loop実行前に以下をセルフチェックする:
1. 各タスクが元の要件に対応しているか（漏れ・スコープ外がないか）
2. セキュリティ: APIキー/SQL/.tomlのルール適合（past-mistakes.md参照）
3. 外部API利用の場合: コスト見積もりを先に出す（past-mistakes.md参照）

エージェント並列起動は「判断が難しい設計上の分岐」がある場合のみ。単純な実装プランには不要。
