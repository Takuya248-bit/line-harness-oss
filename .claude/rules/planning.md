# プランニングルール

## タスクフォーマット（必須）

プランやタスク分解時は「基準が先、手法が後」の順序を厳守する。

```
### タスク名
- 出自: （元の要件/ユーザー指示/調査結果のどれに基づくか）
- Done when: （完了条件。検証可能で曖昧さゼロ）
- Verified by: （検証方法。コマンド、テスト、目視確認等の具体的手段）
- Approach: （実装アプローチ。Done whenを満たすための手順）
```

必ず守ること:
- 全タスクにDone when + Verified by + Approachの3点を含める
- Done whenは検証可能な具体的条件にする（例: 「tsc --noEmitがエラー0で通る」）
- Verified byは確認対象を明示する（例: 「/api/usersのレスポンスが200を返す」）

適用範囲: TaskCreate、プラン・設計文書のタスク分解（progress.mdのログには不要）

## スコープクリープ検出

自動警告トリガー:
1. サブタスク数が15を超えた場合 → 「スコープが膨張していませんか？」と警告 → フェーズ分割を提案
2. 元の要件に遡れないサブタスクが存在する場合 → 「出自が不明です。元の要件のどれに対応しますか？」と確認

スコープ判定基準:
- 必須: ユーザーが明示的に依頼したもの
- 推奨: 依頼を満たすために技術的に必要なもの
- 削除候補: 「ついでにやっておいた方がいい」レベルのもの → ユーザーに提示して判断を仰ぐ

## プラン検証（並列エージェント起動）

以下の条件を満たすプランで実施（軽微な修正は対象外）:
- コード変更が50行以上 / 複数ファイルにまたがる変更 / 新機能追加 / アーキテクチャ影響あり

必須検証エージェント（最低3つ並列起動）:

1. 要件トレーサビリティ: 各タスクが元の要件に対応しているか、漏れ・スコープ外タスクがないか確認
2. 技術健全性: 既存コードパターンとの整合、依存関係、パフォーマンスへの影響
3. セキュリティ: APIキー/SQL/wrangler.tomlのルール適合、入力バリデーション、認証・認可への影響

条件付き追加エージェント:
- コスト検証: 外部APIを利用する場合
- 互換性検証: 既存データやAPIの後方互換性に影響する場合
- UX検証: ユーザー向けUI変更の場合

検証プロセス:
1. プラン完成後、検証エージェントを並列起動
2. 各エージェントが独立に評価し、PASS / CONDITIONAL PASS / FAIL を判定
3. 全PASS → 実装開始 / CONDITIONAL PASS → 条件付きで実装開始 / 1つでもFAIL → プラン修正後に再検証（最大2ラウンド） / 2ラウンド失敗 → ユーザーにエスカレーション

検証エージェントの返答フォーマット:
```
判定: PASS / CONDITIONAL PASS / FAIL
根拠: （判定理由を箇条書き）
指摘事項: （改善すべき点。PASSでも軽微な指摘は可）
```

注意: 検証エージェントは結論のみ返す。検証結果はprogress.mdに記録する。
