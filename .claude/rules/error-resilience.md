# エラーリカバリ・レジリエンスルール（全エージェント必須）

一時的なAPI混雑や失敗は「致命的失敗」ではなく「一時的失敗」として扱い、自動でリカバリする。

## 一時的失敗として扱うエラー

- 529 overloaded_error
- 429 rate_limit_error
- 一時的な network error / timeout
- ストリーム中断

## リトライルール

1. 1回目失敗: 10秒待機 → 再試行
2. 2回目失敗: 30秒待機 → 再試行
3. 3回目失敗: 60秒待機 → 再試行
4. 4回目以降: 処理を一時中断し「完了+再開可能状態」を保存

## 基本方針

1. すぐに諦めない
2. 同じ処理を自動リトライする
3. 毎回少し待ってから再試行する
4. 連続失敗時は待機時間を伸ばす（exponential backoff）
5. 部分的に成功している場合は、そこから続行する
6. 完了済み工程はスキップする（最初からやり直さない）
7. 最後に「どこまで完了したか」「どこで失敗したか」「次に何を再開すべきか」を必ず残す

## 禁止事項

- エラー1回で処理全体を中断しない
- 毎回最初からやり直さない
- 完了済み工程を無駄に再実行しない
- 状況説明だけして止まらない（次のアクションを必ず実行する）

## フェーズ終了時の必須レポート

各フェーズ（大きな作業の区切り）の終了時に以下を出力する:

1. 完了した作業
2. 失敗した作業（原因含む）
3. 保留中の作業
4. 次に再開すべき作業
5. 生成済みファイル一覧
6. 現時点でのベスト成果物

## ログ方針

- 失敗理由を短く progress.md に残す
- 成功した工程も残す
- 再開に必要な情報を必ずまとめる
