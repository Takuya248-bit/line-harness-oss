# プラン・タスクフォーマットルール

プランやタスク分解時は「基準が先、手法が後」の順序を厳守する。

## タスク記述フォーマット（必須）

```
### タスク名
- Done when: （完了条件。検証可能で曖昧さゼロ）
- Verified by: （検証方法。コマンド、テスト、目視確認等の具体的手段）
- Approach: （実装アプローチ。Done whenを満たすための手順）
```

## なぜこの順序か

1. Done when を先に書くことで「何を達成するか」が明確になる
2. Verified by を次に書くことで「どう確認するか」が実装前に決まる
3. Approach は最後。基準が定まってから手法を決める

## 禁止事項

- Approachだけ書いてDone when/Verified byがないタスク
- 「正しく動作すること」のような曖昧なDone when
- 「目視確認」だけのVerified by（何を見るか具体的に書く）

## 適用範囲

- TaskCreate で作成するタスク
- プラン・設計文書のタスク分解
- progress.md の作業ログには不要（ログは結果報告なので）
