# レビューワークフロー（日次・セッション・週次）

## 日次チェック（/ohayou 実行時に秘書が実施）

前日のprogress.mdからサブエージェント起動を数え、以下を1行報告:
- 「昨日のエージェント: Sonnet X回 / Opus Y回 / Haiku Z回」
- Opusで実行されたタスクが「判断・分析系」以外だった場合 → 「○○はSonnetで十分でした」と指摘

## セッションチェック（/snapshot 実行時に追記）

progress.mdのスナップショットに以下を追加:
- モデル使い分け: （そのセッションで起動したサブエージェントのモデル内訳）
- 節約メモ: （Opusで起動したが振り返るとSonnetで十分だったもの。なければ「適切」）

## 週次レビュー

タイミング: 毎週月曜の /ohayou 実行時、または「週次レビュー」「振り返り」とユーザーが言った場合

手順:
1. 過去7日分の progress.md を集計
2. 過去7日分の decisions.md を確認
3. 以下のフォーマットで `secretary/notes/YYYY-WXX-review.md` を作成

レビューフォーマット:
```
# 週次レビュー YYYY-WXX（MM/DD〜MM/DD）

## 今週の実績
- 完了タスク数: X件
- 失敗/リトライ: Y件
- 新規ルール追加: Z件

## 繰り返しパターン（要注意）
- （同じミスや同じ指摘が複数回あったもの）

## フィードバック昇格候補
- （2回以上出たフィードバックでまだルール化されていないもの）

## エージェント品質トレンド
- リサーチ: 改善 / 横ばい / 悪化
- コンテンツ: 改善 / 横ばい / 悪化
- 実装: 改善 / 横ばい / 悪化

## トークン効率
- エージェント起動: Haiku X回 / Sonnet Y回 / Opus Z回
- Opusが過剰だったケース: （あれば具体的に。なければ「なし」）
- CLAUDE.md行数: 現在X行（目標40行以下）
- rules/ファイル数: X個（重複・陳腐化チェック）
- 改善提案: （あれば）

## 来週の改善アクション
- （具体的に1〜3個）
```

ルール:
- レビューファイルは週1つ（同一週は追記）
- 品質トレンドの判断根拠を1行添える
- 改善アクションは実行可能な粒度にする（例: 「○○のテストを追加する」）

## 継続的な最適化トリガー

以下を検出したら秘書が自動提案:
- CLAUDE.mdが50行を超えた → rules/への分離を提案
- 同じモデル指摘が2回 → CLAUDE.mdのモデル使い分け表を更新
- rules/が15ファイルを超えた → 統合・整理を提案
- progress.mdが1日100行を超えた → ログ粒度の見直しを提案
