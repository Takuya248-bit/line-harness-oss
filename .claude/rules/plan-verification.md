# プラン段階での並列検証ルール

実装プラン作成後、コードを書く前に検証エージェントを並列起動する。
「実装してから気づく」を撲滅するための仕組み。

## 検証トリガー

以下の条件を満たすプランに対して検証を実施する:
- コード変更が50行以上見込まれるもの
- 複数ファイルにまたがる変更
- 新機能の追加
- アーキテクチャに影響する変更

軽微な修正（typo、設定値変更、1ファイル内の小修正）は対象外。

## 検証エージェント（最低3つを並列起動）

### 必須エージェント

1. 要件トレーサビリティ
   - プランの各タスクがユーザーの元の要件に対応しているか確認
   - 漏れている要件がないか確認
   - スコープ外のタスクが紛れていないか確認

2. 技術健全性
   - 既存コードベースのパターンと整合しているか
   - 依存関係に問題がないか
   - パフォーマンスへの影響

3. セキュリティ
   - CLAUDE.mdのセキュリティルール（APIキー、SQL、wrangler.toml）との整合
   - 入力バリデーション
   - 認証・認可への影響

### 条件付きエージェント（該当時のみ追加）

- コスト検証: 外部APIを利用する場合（コストチェックルール適用）
- 互換性検証: 既存データやAPIの後方互換性に影響する場合
- UX検証: ユーザー向けUI変更の場合

## 検証プロセス

1. プラン完成後、検証エージェントを並列起動
2. 各エージェントが独立に評価し、PASS / CONDITIONAL PASS / FAIL を判定
3. 結果を統合:
   - 全PASS → 実装開始
   - CONDITIONAL PASS → 条件付きで実装開始（条件を明示）
   - 1つでもFAIL → プラン修正後に再検証（最大2ラウンド）
   - 2ラウンド失敗 → ユーザーにエスカレーション

## 検証エージェントの返答フォーマット

```
判定: PASS / CONDITIONAL PASS / FAIL
根拠: （判定理由を箇条書き）
指摘事項: （改善すべき点。PASSでも軽微な指摘は可）
```

## 注意

- 検証エージェントは結論のみ返す（context-efficiency.md準拠）
- 検証結果はprogress.mdに記録する
