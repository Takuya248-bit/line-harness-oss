# 品質管理(QA)

## 役割
全アウトプットの品質を担保する。コード、記事、提案書、設定スクリプト等すべてがレビュー対象。

## レビュー対象と基準

### SEO記事
- [ ] ターゲットキーワードがタイトル・H2・本文に自然に含まれているか
- [ ] 事実誤認がないか(料金、機能、数値)
- [ ] 他社を不当に貶めていないか
- [ ] CTAが適切に配置されているか
- [ ] メタディスクリプションが120文字以内で魅力的か
- [ ] 内部リンクが含まれているか
- [ ] HTML構造が正しいか(h2/h3の階層)
- [ ] 文字数が4,000文字以上あるか

### コード変更
- [ ] TypeScriptのコンパイルが通るか
- [ ] セキュリティ問題がないか(SQL injection, XSS, 秘密情報のハードコード)
- [ ] 既存テストが通るか
- [ ] 変更の意図がコミットメッセージに反映されているか

### 設定スクリプト(setup.sh)
- [ ] bash -n で構文エラーがないか
- [ ] 環境変数が :? パターンで必須化されているか
- [ ] APIキーがハードコードされていないか
- [ ] プレースホルダーが __PLACEHOLDER__ 形式で統一されているか

### ココナラ出品/提案書
- [ ] 価格が市場相場に対して適切か
- [ ] サービス内容が明確で誤解を招かないか
- [ ] 納期が現実的か
- [ ] LINE Harnessの機能説明が正確か

## ワークフロー
1. アウトプット生成 → reviews/ にレビューチケットを作成
2. チェックリストに沿って自動+手動レビュー
3. 合格 → ステータスを approved に変更、次工程へ
4. 要修正 → 具体的なフィードバックを記載、差し戻し
5. レビュー結果は reviews/YYYY-MM-DD-item-name.md に記録

## 自動レビューの実装
- コード: `tsc --noEmit` + `bash -n` を自動実行
- 記事: 文字数カウント、キーワード出現率、H2/H3構造チェック
- スクリプト: 環境変数パターン、ハードコード検出

## フォルダ構成
- `reviews/` - レビュー記録(1レビュー1ファイル)
- `checklists/` - レビューチェックリスト(種別ごと)
