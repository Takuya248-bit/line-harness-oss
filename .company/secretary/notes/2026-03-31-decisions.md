
## 19:28 Lstep v3 フロー設計決定

### フェーズ主導設計の採用
- 決定: シナリオ従・フェーズ主導の設計を採用
- フェーズ順: 01(登録)→02(プラン興味)→03(チャット相談)→06(見積)→08(面談参加)→10(成約)、99(休眠)
- CVは温度感順に優先。上位フェーズに進んだら下位シナリオを停止する

### S_01完了時の動作
- 決定: S_05を開始しない。カルーセル送信のみ
- 理由: S_05 Day0とカルーセルの同時送信を防ぐ
- カルーセル無反応時は配信空白を許容（メニューからの自発行動待ち）

### S_02_xxx完了時の遷移
- 決定: 7本全てのDay4にアクション追加（フェーズ99+S_05開始+メニュー_05）
- 理由: 既存の「自動_シナリオ02完了」はS_02_xxxには発火しない

### フェーズ03（チャット相談）
- 決定: 例外ゾーン。現シナリオを停止しない。S_chatを併走させる
- 理由: チャットが不発に終わった場合の配信停止を防ぐ

### スコープ外

- ブロック解除: 旧のまま

## 19:49 Notion AI × Claude Code 棲み分け決定

### ツール棲み分け方針
- 決定: 日常業務はNotion AI、専門的生成物・SaaS連携・自動化はClaude Code
- Notion AIの用途: 返信文、タスク管理、議事録、画像生成、Skill、@呼び出し
- Claude Codeの用途: Remotion動画、PowerPointスライド、Playwright自動操作、Chrome経由のWeb操作、コード実装

### Lstep入力のNotion連携
- 決定: Notion(コンテンツ管理) → notion-bridge.mjs(YAML変換) → Playwright(自動操作) の3層構成を採用
- 理由: コンテンツ準備・確認のトークン消費を削減、Notionテーブルの一覧性、プロパティ型制約によるミス防止、非エンジニアもシナリオ編集可能
- 実装: lstep-automation/src/notion-bridge.mjs 作成済み、Notion DB設計書 docs/notion-setup.md 作成済み

## 20:50 画像生成アーキテクチャ決定
- 決定: Satori+resvg（Workers内）→ Puppeteer+HTML/CSS（GitHub Actions）に移行
- 経緯: Workers無料プランのCPU制限(10ms)でSatori PNG変換が失敗 → Sharp+SVGに移行するもクオリティ不足 → Puppeteer+HTML/CSSが最終形
- コスト: GitHub Actions無料枠(2000分/月)内
